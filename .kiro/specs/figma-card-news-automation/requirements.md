# 요구사항 문서

## 소개

U+ IPTV에 노출되는 영화/드라마 홍보용 카드뉴스 이미지를 자동 생성하는 시스템이다. 비개발자(디자이너)가 웹 UI에서 버튼 클릭만으로 구글 스프레드시트의 데이터를 읽고, Figma API(REST API + Plugin API 하이브리드)를 통해 카드뉴스 이미지를 렌더링하여 JPG로 다운로드할 수 있도록 한다. 로컬 PC 환경에서 브라우저를 통해 동작하며, 팀 내 개발자 없이 디자이너가 독립적으로 운영할 수 있도록 설계한다.

## 용어 정의

- **시스템**: Figma 카드뉴스 자동화 시스템 전체 (웹 UI + 백엔드 서버)
- **웹_UI**: 사용자가 브라우저에서 접속하여 카드뉴스 생성을 제어하는 프론트엔드 인터페이스
- **스프레드시트_리더**: 구글 스프레드시트에서 카드뉴스 데이터를 읽어오는 모듈
- **드라이브_리더**: 구글 드라이브에서 배경 이미지와 로고 파일을 읽어오는 모듈
- **Figma_렌더러**: Figma REST API와 Plugin API를 하이브리드로 활용하여 템플릿 프레임을 복제하고, 레이어를 수정하고, 이미지를 렌더링하는 모듈. 텍스트 수정은 REST API, 이미지(배경/로고) 교체는 Plugin API를 사용한다.
- **카드뉴스**: 808x454px 크기의 영화/드라마 홍보용 JPG 이미지
- **템플릿_프레임**: Figma 파일에 미리 만들어진 카드뉴스 레이아웃 프레임. 시스템이 인식할 수 있도록 약속된 레이어 이름 규칙을 따라야 한다.
- **레이어_이름_규칙**: Figma 템플릿 내 동적 레이어를 시스템이 식별하기 위한 이름 규칙. 배경(`bg_image`), 로고(`logo`), 기본문구(`main_text`), 추가문구(`sub_text`), 카피라이트(`copyright`), 뱃지 컨테이너(`badge_container`) 등 약속된 이름을 사용한다.
- **복제_프레임**: 카드뉴스 생성 시 템플릿_프레임을 복제(clone)하여 만든 개별 프레임. 각 카드뉴스마다 독립된 복제_프레임이 생성된다.
- **뱃지**: 카드뉴스에 표시되는 20종의 태그 (무료, UHD, AI 보이스 등)
- **뱃지_Variant**: 노출 뱃지 개수(1~4개)에 따라 전환되는 Figma 컴포넌트 변형
- **기본_문구**: 영화/드라마 홍보를 위한 1~3줄 짧은 문장
- **추가_문구**: 수상내역 등 선택적으로 노출되는 부가 정보 텍스트
- **카피라이트**: 콘텐츠별 저작권 표기 텍스트 (선택적, 빈칸 가능)
- **행**: 스프레드시트의 한 행으로, 카드뉴스 1장에 대응하는 데이터 단위
- **배치**: 1회 생성 실행에서 처리되는 행 묶음 (1~30장)
- **설정_마법사**: 시스템 최초 실행 시 API 인증 등 필수 설정을 단계별로 안내하는 웹 UI 화면
- **히스토리_탭**: 구글 스프레드시트 내 별도 탭으로, 카드뉴스 생성 이력을 기록하는 시트

## 요구사항

### 요구사항 1: 스프레드시트 데이터 읽기

**사용자 스토리:** 디자이너로서, 구글 스프레드시트에 작성한 카드뉴스 정보를 시스템이 자동으로 읽어오길 원한다. 이를 통해 수동 데이터 입력 없이 카드뉴스를 생성할 수 있다.

#### 인수 조건

1. WHEN 사용자가 "생성" 버튼을 클릭하면, THE 스프레드시트_리더 SHALL 구글 스프레드시트에서 모든 행 데이터를 읽어온다.
2. THE 스프레드시트_리더 SHALL 각 행에서 영화제목, 배경파일명, 로고파일명, 기본문구, 추가문구, 뱃지1~4, 카피라이트 컬럼을 파싱한다.
3. WHEN 추가문구, 뱃지 컬럼, 또는 카피라이트 컬럼이 빈칸이면, THE 스프레드시트_리더 SHALL 해당 필드를 빈 값으로 처리하고 정상 진행한다.
4. THE 스프레드시트_리더 SHALL 셀 내 줄바꿈(Alt+Enter)을 보존하여 파싱한다.
5. FOR ALL 유효한 스프레드시트 행에 대해, 파싱 후 다시 동일 형식으로 직렬화하면 원본 데이터와 동일한 결과를 생성한다 (라운드트립 속성).

### 요구사항 2: 이미지 소스 로드

**사용자 스토리:** 디자이너로서, 구글 드라이브에 업로드한 배경 이미지와 로고를 시스템이 자동으로 가져오길 원한다. 이를 통해 파일을 수동으로 옮기지 않아도 된다.

#### 인수 조건

1. WHEN 스프레드시트 행의 배경파일명이 주어지면, THE 드라이브_리더 SHALL 구글 드라이브의 `카드뉴스_소스/배경이미지/` 폴더에서 해당 파일을 다운로드한다.
2. WHEN 스프레드시트 행의 로고파일명이 주어지면, THE 드라이브_리더 SHALL 구글 드라이브의 `카드뉴스_소스/로고/` 폴더에서 해당 파일을 다운로드한다.
3. IF 지정된 파일명이 구글 드라이브 폴더에 존재하지 않으면, THEN THE 드라이브_리더 SHALL 해당 행을 오류로 기록하고 다음 행 처리를 계속한다.

### 요구사항 3: Figma 템플릿 복제 및 레이어 수정

**사용자 스토리:** 디자이너로서, Figma 템플릿의 동적 요소(배경, 로고, 문구, 뱃지, 카피라이트)가 스프레드시트 데이터에 맞게 자동으로 교체되길 원한다. 이를 통해 수작업 편집 시간을 절약할 수 있다.

#### 인수 조건

1. WHEN 배치 생성이 시작되면, THE Figma_렌더러 SHALL Figma REST API를 통해 템플릿_프레임의 컴포넌트 스펙(위치, 사이즈, 폰트)을 1회 읽어와 배치 내에서 캐싱한다.
2. WHEN 각 카드뉴스 생성이 시작되면, THE Figma_렌더러 SHALL 템플릿_프레임을 복제(clone)하여 독립된 복제_프레임을 생성한다.
3. WHEN 배경 이미지 데이터가 준비되면, THE Figma_렌더러 SHALL Figma Plugin API를 통해 복제_프레임의 배경 이미지 레이어를 해당 이미지로 교체한다.
4. WHEN 로고 데이터가 준비되면, THE Figma_렌더러 SHALL Figma Plugin API를 통해 복제_프레임의 로고 레이어를 해당 로고로 교체한다.
5. WHEN 기본문구가 주어지면, THE Figma_렌더러 SHALL Figma REST API를 통해 복제_프레임의 기본문구 텍스트 레이어를 해당 텍스트로 수정한다.
6. WHEN 추가문구가 존재하면, THE Figma_렌더러 SHALL 복제_프레임의 추가문구 텍스트 레이어를 해당 텍스트로 수정한다.
7. WHEN 추가문구가 빈 값이면, THE Figma_렌더러 SHALL 복제_프레임의 추가문구 텍스트 레이어를 숨김 처리한다.
8. WHEN 카피라이트 텍스트가 주어지면, THE Figma_렌더러 SHALL 복제_프레임의 카피라이트 레이어를 해당 텍스트로 수정한다.
9. WHEN 카피라이트가 빈 값이면, THE Figma_렌더러 SHALL 복제_프레임의 카피라이트 레이어를 숨김 처리한다.

### 요구사항 4: 뱃지 처리

**사용자 스토리:** 디자이너로서, 스프레드시트에 입력한 뱃지 이름에 따라 카드뉴스에 올바른 뱃지가 올바른 위치에 표시되길 원한다. 이를 통해 뱃지 배치를 수동으로 조정하지 않아도 된다.

#### 인수 조건

1. THE Figma_렌더러 SHALL 20종 뱃지(무료, UHD, AI 보이스, 사전예약, 할인, 프리미엄무료, 가격인하, 소장, HD, 우리말, 예고편, 가치봄-자막+수어, Dolby ATMOS, Dolby VISION, Dolby VISION-ATMOS, 이벤트, U+독점, U+오리지널, U+STAGE, 유플레이)를 인식한다.
2. WHEN 뱃지 컬럼에 유효한 뱃지 이름이 1~4개 입력되면, THE Figma_렌더러 SHALL 해당 개수에 맞는 뱃지_Variant로 전환한다.
3. THE Figma_렌더러 SHALL 각 뱃지를 뱃지_Variant에 정의된 고정 위치에 배치한다.
4. THE 시스템 SHALL 뱃지 이름 비교 시 대소문자 및 앞뒤 공백을 정규화하여 매칭한다 (예: `dolby vision` → `Dolby VISION`).
5. IF 뱃지 컬럼에 20종에 포함되지 않는 이름이 입력되면, THEN THE 시스템 SHALL 해당 행을 오류로 기록하고 다음 행 처리를 계속한다.
6. WHEN 모든 뱃지 컬럼이 빈칸이면, THE Figma_렌더러 SHALL 뱃지 레이어를 숨김 처리한다.

### 요구사항 5: 이미지 렌더링 및 내보내기

**사용자 스토리:** 디자이너로서, Figma에서 렌더링된 카드뉴스를 규격에 맞는 JPG 파일로 자동 다운로드하길 원한다. 이를 통해 별도의 이미지 편집 없이 바로 사용할 수 있다.

#### 인수 조건

1. WHEN 복제_프레임의 레이어 수정이 완료되면, THE Figma_렌더러 SHALL Figma REST API의 /images 엔드포인트를 호출하여 해당 복제_프레임의 렌더링된 이미지를 요청한다.
2. THE Figma_렌더러 SHALL 렌더링된 이미지를 808x454px 크기의 JPG 형식으로 내보낸다.
3. THE 시스템 SHALL 먼저 JPG 품질 80%로 내보내기를 시도한다.
4. IF 80% 품질에서 파일 용량이 500KB를 초과하면, THEN THE 시스템 SHALL 품질을 점진적으로 낮추어 500KB 이하가 될 때까지 재시도한다 (최소 품질: 70%).
5. IF 70% 품질에서도 500KB를 초과하면, THEN THE 시스템 SHALL 해당 파일을 70% 품질로 저장하고 웹_UI에 용량 초과 경고를 표시한다 (500KB는 권장 기준이므로 생성은 완료한다).
6. THE 시스템 SHALL 파일명을 `카드뉴스_(영화제목).jpg` 형식으로 지정한다.
7. IF 동일한 영화제목의 파일이 이미 존재하면, THEN THE 시스템 SHALL 두 번째부터 `카드뉴스_(영화제목)_02.jpg`, `카드뉴스_(영화제목)_03.jpg` 형식으로 순번을 추가한다.
8. THE 시스템 SHALL 생성된 파일을 구글 드라이브의 `카드뉴스_소스/결과물/` 폴더에 업로드한다.

### 요구사항 6: 텍스트 오버플로우 방지

**사용자 스토리:** 디자이너로서, 입력한 텍스트가 Figma 레이아웃 영역을 넘치지 않도록 보장되길 원한다. 이를 통해 잘린 텍스트 없이 깔끔한 카드뉴스를 생성할 수 있다.

#### 인수 조건

1. WHEN 복제_프레임에 텍스트가 입력된 후, THE Figma_렌더러 SHALL Figma Plugin API를 통해 해당 텍스트 노드의 absoluteBoundingBox와 부모 프레임의 영역을 비교하여 오버플로우를 검증한다.
2. IF 기본문구 또는 추가문구 텍스트가 해당 레이어의 바운딩 박스를 초과하면, THEN THE 시스템 SHALL 해당 행을 오류로 기록하고 사용자에게 텍스트 수정을 안내한다.
3. THE 시스템 SHALL 텍스트 오버플로우 검증을 이미지 렌더링 전에 수행한다 (Figma에 텍스트 입력 후, 렌더링 요청 전 사후 검증 방식).
4. WHEN 기본문구의 줄 수가 3줄을 초과하면, THE 시스템 SHALL 해당 행을 오류로 기록한다.

### 요구사항 7: 웹 UI - 생성 및 다운로드

**사용자 스토리:** 디자이너로서, 웹 브라우저에서 버튼 클릭만으로 카드뉴스를 생성하고 다운로드하길 원한다. 이를 통해 개발 지식 없이도 카드뉴스를 제작할 수 있다.

#### 인수 조건

1. THE 웹_UI SHALL "생성" 버튼을 제공한다.
2. WHEN 사용자가 "생성" 버튼을 클릭하면, THE 시스템 SHALL 스프레드시트 읽기, 이미지 소스 로드, Figma 템플릿 복제, 레이어 수정, 렌더링, JPG 내보내기를 순차적으로 실행한다.
3. WHEN 이미지 생성이 완료되면, THE 웹_UI SHALL 생성된 카드뉴스 파일을 다운로드할 수 있는 링크를 제공한다.
4. WHILE 이미지 생성이 진행 중이면, THE 웹_UI SHALL 현재 진행 상태(처리 중인 행 번호, 전체 행 수)를 표시한다.
5. THE 시스템 SHALL 1회 실행에 1~30장의 카드뉴스를 생성할 수 있다.

### 요구사항 8: 웹 UI - 미리보기

**사용자 스토리:** 디자이너로서, 카드뉴스를 최종 생성하기 전에 미리보기로 결과를 확인하길 원한다. 이를 통해 오류를 사전에 발견하고 수정할 수 있다.

#### 인수 조건

1. THE 웹_UI SHALL 스프레드시트 데이터를 기반으로 카드뉴스 미리보기 기능을 제공한다.
2. WHEN 사용자가 특정 행의 미리보기를 요청하면, THE 시스템 SHALL 해당 행의 데이터로 Figma 템플릿을 복제하고 레이어를 수정한 뒤, 저해상도(0.5x scale)로 렌더링하여 웹_UI에 표시한다.
3. THE 웹_UI SHALL 미리보기 이미지가 최종 출력물과 레이아웃은 동일하되 해상도가 낮을 수 있음을 안내한다.

### 요구사항 9: 웹 UI - 개별 행 선택 생성

**사용자 스토리:** 디자이너로서, 스프레드시트의 특정 행만 선택하여 카드뉴스를 생성하길 원한다. 이를 통해 수정이 필요한 항목만 재생성할 수 있다.

#### 인수 조건

1. THE 웹_UI SHALL 스프레드시트의 각 행을 개별 선택할 수 있는 체크박스를 제공한다.
2. WHEN 사용자가 특정 행을 선택하고 "생성" 버튼을 클릭하면, THE 시스템 SHALL 선택된 행에 대해서만 카드뉴스를 생성한다.
3. THE 웹_UI SHALL 전체 선택 및 전체 해제 기능을 제공한다.

### 요구사항 10: 웹 UI - 스프레드시트 편집

**사용자 스토리:** 디자이너로서, 웹 UI에서 스프레드시트 데이터를 직접 수정하길 원한다. 이를 통해 구글 스프레드시트와 웹 UI를 번갈아 사용하지 않아도 된다.

#### 인수 조건

1. THE 웹_UI SHALL 스프레드시트 데이터를 테이블 형태로 표시한다.
2. WHEN 사용자가 테이블의 셀을 클릭하면, THE 웹_UI SHALL 해당 셀을 편집 가능한 상태로 전환한다.
3. WHEN 사용자가 셀 편집을 완료하면, THE 웹_UI SHALL 변경된 데이터를 구글 스프레드시트에 동기화한다 (웹 UI → 구글 시트 단방향).
4. WHEN 구글 스프레드시트에서 직접 데이터가 변경된 경우, THE 웹_UI SHALL "새로고침" 버튼을 통해 최신 데이터를 다시 불러오는 기능을 제공한다.
5. THE 웹_UI SHALL 편집 중인 셀에서 줄바꿈 입력을 지원한다.
6. THE 웹_UI SHALL 뱃지 컬럼 편집 시 20종 뱃지 목록에서 자동완성/드롭다운 선택을 제공한다.
7. WHEN 뱃지 컬럼에 유효하지 않은 이름이 입력되면, THE 웹_UI SHALL 실시간으로 오류를 표시한다 (빨간 테두리 등).

### 요구사항 11: 웹 UI - 히스토리 관리

**사용자 스토리:** 디자이너로서, 이전에 생성한 카드뉴스 이력을 확인하고 다시 다운로드하길 원한다. 이를 통해 과거 작업물을 쉽게 찾을 수 있다.

#### 인수 조건

1. THE 시스템 SHALL 카드뉴스 생성 이력을 구글 스프레드시트의 히스토리_탭에 기록한다.
2. THE 히스토리_탭 SHALL 각 생성 기록에 생성 일시, 생성된 카드뉴스 수, 오류 수, 생성된 파일명 목록을 포함한다.
3. THE 웹_UI SHALL 히스토리_탭의 데이터를 읽어 카드뉴스 생성 히스토리를 목록으로 표시한다.
4. WHEN 사용자가 히스토리 항목을 선택하면, THE 웹_UI SHALL 해당 생성 결과의 카드뉴스 이미지 목록과 오류 목록을 표시한다.
5. WHEN 사용자가 히스토리의 카드뉴스 이미지를 선택하면, THE 웹_UI SHALL 구글 드라이브 `카드뉴스_소스/결과물/` 폴더에서 해당 이미지를 다시 다운로드할 수 있는 기능을 제공한다.

### 요구사항 12: 에러 처리 및 보고

**사용자 스토리:** 디자이너로서, 카드뉴스 생성 중 오류가 발생해도 나머지 카드뉴스는 정상 생성되길 원한다. 이를 통해 오류 하나 때문에 전체 작업을 다시 시작하지 않아도 된다.

#### 인수 조건

1. IF 특정 행에서 오류가 발생하면(뱃지 이름 오타, 파일 미존재, 텍스트 오버플로우 등), THEN THE 시스템 SHALL 해당 행을 건너뛰고 나머지 행의 카드뉴스 생성을 계속한다.
2. WHEN 카드뉴스 생성이 완료되면, THE 웹_UI SHALL 오류가 발생한 행 목록과 각 오류 원인을 표시한다.
3. THE 웹_UI SHALL 성공한 카드뉴스 수와 실패한 카드뉴스 수를 요약하여 표시한다.
4. IF Figma API 호출이 실패하면, THEN THE 시스템 SHALL 오류 메시지를 웹_UI에 표시하고 재시도 옵션을 제공한다.
5. IF 구글 스프레드시트 또는 구글 드라이브 연결이 실패하면, THEN THE 시스템 SHALL 연결 오류 메시지를 웹_UI에 표시한다.
6. WHEN Figma API Rate Limit에 도달하면, THE 시스템 SHALL 자동으로 대기(backoff)한 후 재시도한다.
7. WHILE Figma API Rate Limit 대기 중이면, THE 웹_UI SHALL "API 대기 중" 상태를 표시한다.
8. THE 시스템 SHALL Figma API 호출 간 적절한 딜레이(1~2초)를 삽입하여 Rate Limit 발생을 최소화한다.

### 요구사항 13: Figma 프레임 관리

**사용자 스토리:** 디자이너로서, 카드뉴스 생성으로 Figma 파일에 쌓인 복제 프레임을 쉽게 정리하길 원한다. 이를 통해 Figma 파일이 무거워지는 것을 방지할 수 있다.

#### 인수 조건

1. THE 시스템 SHALL 카드뉴스 생성 시 복제된 프레임을 Figma 파일에 유지하여 디자이너가 직접 확인할 수 있도록 한다.
2. THE 웹_UI SHALL "프레임 정리" 버튼을 제공한다.
3. WHEN 사용자가 "프레임 정리" 버튼을 클릭하면, THE 시스템 SHALL 이전 배치에서 생성된 복제_프레임 목록을 표시한다.
4. WHEN 사용자가 정리할 배치를 선택하면, THE Figma_렌더러 SHALL 해당 배치의 복제_프레임을 Figma 파일에서 삭제한다.

### 요구사항 14: 로컬 실행 환경 및 초기 설정

**사용자 스토리:** 디자이너로서, 별도의 서버 설정 없이 로컬 PC에서 시스템을 실행하길 원한다. 개발자 도움 없이 초기 설정부터 독립적으로 수행할 수 있어야 한다.

#### 인수 조건

1. THE 시스템 SHALL 로컬 PC에서 웹 서버를 실행하여 브라우저로 접속할 수 있는 환경을 제공한다.
2. THE 시스템 SHALL 라이선스 문제가 없는 로컬 폰트를 사용한다.
3. THE 시스템 SHALL Google Service Account 방식으로 구글 스프레드시트 및 구글 드라이브에 인증한다.
4. WHEN 시스템이 최초 실행되면, THE 웹_UI SHALL 설정_마법사를 표시한다.
5. THE 설정_마법사 SHALL 다음 단계를 스크린샷 포함 상세 가이드와 함께 안내한다: (a) Google Cloud Console 프로젝트 생성, (b) Google Sheets API 및 Google Drive API 활성화, (c) 서비스 계정 생성 및 JSON 키 파일 다운로드, (d) 스프레드시트/드라이브 폴더를 서비스 계정 이메일에 공유, (e) Figma API 토큰 발급.
6. THE 설정_마법사 SHALL JSON 키 파일을 드래그앤드롭으로 업로드할 수 있는 UI를 제공한다.
7. THE 설정_마법사 SHALL 구글 스프레드시트 URL 입력 시 자동으로 스프레드시트 ID를 추출한다.
8. THE 설정_마법사 SHALL 구글 드라이브 폴더 URL 입력 시 자동으로 폴더 ID를 추출한다.
9. THE 설정_마법사 SHALL Figma API 토큰 입력 필드를 제공한다.
10. THE 설정_마법사 SHALL Figma 파일 URL 입력 필드를 제공하고, URL에서 자동으로 File Key를 추출한다.
11. WHEN Figma File Key가 확인되면, THE 설정_마법사 SHALL Figma REST API를 통해 해당 파일의 최상위 프레임 목록을 조회하여 드롭다운으로 표시한다.
12. THE 설정_마법사 SHALL 사용자가 드롭다운에서 템플릿_프레임을 선택할 수 있도록 한다.
13. WHEN 템플릿_프레임이 선택되면, THE 설정_마법사 SHALL 해당 프레임 내 레이어 목록을 조회하여 레이어_이름_규칙(`bg_image`, `logo`, `main_text`, `sub_text`, `copyright`, `badge_container`)과 일치하는지 검증한다.
14. IF 필수 레이어 이름이 누락되면, THEN THE 설정_마법사 SHALL 누락된 레이어 이름 목록과 Figma 템플릿 수정 안내를 표시한다.
15. THE 설정_마법사 SHALL Figma 템플릿 레이어 이름 규칙 안내 문서를 제공한다 (각 레이어의 역할과 필수 이름을 스크린샷과 함께 설명).
16. THE 설정_마법사 SHALL "연결 테스트" 버튼을 제공하여 모든 API(구글 스프레드시트, 구글 드라이브, Figma)의 연결 상태 및 템플릿 프레임 유효성을 검증한다.
17. IF 연결 테스트에서 실패한 항목이 있으면, THEN THE 설정_마법사 SHALL 실패 원인과 해결 방법을 구체적으로 안내한다.
18. WHEN 모든 설정이 완료되면, THE 시스템 SHALL 설정 정보를 로컬 환경 설정 파일에 저장한다.
19. WHEN 시스템이 시작되면, THE 시스템 SHALL 저장된 설정의 유효성을 검증하고, 유효하지 않으면 설정_마법사를 다시 표시한다.
