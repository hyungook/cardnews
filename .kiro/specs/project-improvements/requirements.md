# 프로젝트 개선 요구사항

## 개요
카드뉴스 자동화 프로젝트의 보안, 안정성, 사용성을 개선하여 외부 접근 가능한 서비스로 전환합니다.

## 핵심 요구사항

### 1. 보안 및 접근 제어
**Req 1.1**: 외부 접근 가능하도록 CORS 설정 개선
**Req 1.2**: 파일 업로드/삭제 시 경로 탐색 공격 방지
**Req 1.3**: 업로드된 파일만 삭제 가능하도록 화이트리스트 검증
**Req 1.4**: 에러 메시지 표준화 및 민감 정보 노출 방지

### 2. 파일 관리
**Req 2.1**: 중복 파일 업로드 방지 (파일 해시 기반)
**Req 2.2**: 동시 업로드 제한 (로고 + 배경 합쳐서 40개)
**Req 2.3**: 파일명 충돌 시 자동 번호 부여 (예: `영화_1.jpg`)
**Req 2.4**: 업로드 전 기존 파일 덮어쓰기 확인
**Req 2.5**: 파일명 수정 기능 추가

### 3. 자동 삭제 관리
**Req 3.1**: 30일 이상 된 결과물 목록 조회 API
**Req 3.2**: 삭제 예정 파일 미리보기 기능
**Req 3.3**: 수동 삭제 실행 기능

### 4. API 연결 테스트
**Req 4.1**: Google Sheets API 실제 연결 테스트
**Req 4.2**: Figma API 실제 연결 테스트
**Req 4.3**: 연결 실패 시 구체적인 에러 메시지 제공

### 5. 로깅 시스템
**Req 5.1**: Winston 로깅 라이브러리 도입
**Req 5.2**: 로그 레벨 구분 (debug, info, warn, error)
**Req 5.3**: 로그 파일 저장 (일별 로테이션)
**Req 5.4**: 프로덕션 환경에서 debug 로그 비활성화

### 6. 환경 변수 관리
**Req 6.1**: `.env` 파일 지원
**Req 6.2**: 개발/프로덕션 환경 구분
**Req 6.3**: 주요 설정값 환경 변수화
  - `PORT`: 서버 포트 (기본값: 3000)
  - `NODE_ENV`: 환경 (development/production)
  - `BASE_DIR`: 이미지 저장 경로 (기본값: ./images)
  - `MAX_FILE_SIZE`: 최대 파일 크기 (기본값: 20MB)
  - `MAX_CONCURRENT_UPLOADS`: 동시 업로드 제한 (기본값: 40)
  - `CLEANUP_DAYS`: 자동 삭제 기간 (기본값: 30)
  - `LOG_LEVEL`: 로그 레벨 (기본값: info)

### 7. API 응답 표준화
**Req 7.1**: 성공 응답 형식 통일
```typescript
{
  success: true,
  data: { ... }
}
```

**Req 7.2**: 에러 응답 형식 통일
```typescript
{
  success: false,
  error: {
    code: "ERROR_CODE",
    message: "사용자 친화적 메시지",
    details?: "개발 환경에서만 제공"
  }
}
```

**Req 7.3**: HTTP 상태 코드 일관성
- 200: 성공
- 400: 잘못된 요청
- 404: 리소스 없음
- 409: 충돌 (중복 파일 등)
- 429: 요청 제한 초과
- 500: 서버 오류

### 8. 테스트 개선
**Req 8.1**: API 엔드포인트 통합 테스트 추가
**Req 8.2**: 파일 업로드/삭제 시나리오 테스트
**Req 8.3**: 에러 처리 테스트
**Req 8.4**: 테스트 커버리지 70% 이상

### 9. 성능 최적화
**Req 9.1**: 파일 목록 조회 시 페이지네이션 (기본 50개)
**Req 9.2**: 이미지 업로드 시 스트리밍 처리
**Req 9.3**: 메모리 사용량 모니터링

### 10. 문서화
**Req 10.1**: API 문서 자동 생성 (Swagger/OpenAPI)
**Req 10.2**: 설치 가이드 작성
**Req 10.3**: 배포 가이드 작성
**Req 10.4**: 에러 코드 문서 작성

### 11. 배포 및 운영
**Req 11.1**: Dockerfile 작성
**Req 11.2**: docker-compose.yml 작성
**Req 11.3**: 헬스 체크 엔드포인트 (`/health`)
**Req 11.4**: PM2 설정 파일 작성
**Req 11.5**: 프로덕션 빌드 스크립트

## 우선순위

### Phase 1: 보안 및 안정성 (필수)
- Req 1.2, 1.3, 1.4: 보안 취약점 수정
- Req 2.1, 2.2, 2.3: 파일 관리 개선
- Req 5.1, 5.2, 5.3: 로깅 시스템
- Req 6.1, 6.2, 6.3: 환경 변수 관리
- Req 7.1, 7.2, 7.3: API 표준화

### Phase 2: 기능 개선 (중요)
- Req 2.4, 2.5: 파일명 관리
- Req 3.1, 3.2, 3.3: 자동 삭제 관리
- Req 4.1, 4.2, 4.3: API 연결 테스트
- Req 9.1, 9.2: 성능 최적화

### Phase 3: 운영 및 문서화 (권장)
- Req 8.1, 8.2, 8.3: 테스트 개선
- Req 10.1, 10.2, 10.3, 10.4: 문서화
- Req 11.1, 11.2, 11.3, 11.4, 11.5: 배포 설정

## 제외 사항
- 설정 파일 백업/복원 기능 (단순 작업이므로 불필요)
- 사용자 인증 시스템 (민감 정보 없음)
- 복잡한 모니터링 시스템 (초기 단계에서는 불필요)
